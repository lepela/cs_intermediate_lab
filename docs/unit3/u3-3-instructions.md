## 개념

지침(Instructions)은 에이전트에게 주는 **자연어 행동 규칙**입니다. "너는 누구이고, 무엇을 어떻게 답하고, 무엇은 하지 말라"를 적습니다.

이번 유닛에서는 **기초 블록**만 작성합니다 — 아직 도구가 없으니 "조회/평가" 같은 동작 규칙은 다음 유닛에서 도구를 붙일 때 추가합니다.

{: .important }
**지침은 정적 산출물이 아니라 시스템이 자라며 함께 진화합니다.** 이 과정의 핵심 교수법은 *테스트 → 실패 관찰 → 지침 블록 추가 → 재테스트*의 반복입니다. 예컨대 Unit 4에서 "거부된 지원자가 조회에 샌다"를 보고 비노출 규칙을, "숫자 코드 0을 못 읽는다"를 보고 매핑 규칙을 덧붙이게 됩니다. 지금은 그 출발점인 기초만 둡니다.

---

## 단계별 가이드

### 1단계. 역할·페르소나 작성

지침 칸에 에이전트의 정체성과 말투를 적습니다.

```
너는 누리커머스(주)의 채용을 돕는 "면접관 에이전트"다.
채용 담당자가 지원자를 조회·평가하고 면접을 준비하도록 돕는다.
답변은 한국어로, 간결하고 정중하게 한다.
근거가 되는 기준(채용 가이드·직무기술서·회사 지침·평가 루브릭)에
입각해 답하고, 기준에 없는 내용은 단정하지 않는다.
```

<!-- SCREENSHOT: u3-s07 — 지침: 역할·페르소나 -->

### 2단계. 역할 경계 규칙

에이전트가 **무엇을 어디서 찾아야 하는지** 경계를 명시합니다. 이것이 다음 유닛에서 "기준 질문엔 답하고 데이터 질문엔 막히는" 행동의 뿌리가 됩니다.

```
[자료의 역할]
- Knowledge(채용 가이드·JD·회사 지침·평가 루브릭) = 평가의 "기준".
  "백엔드 채용 기준이 뭐야", "적합도는 어떻게 평가해", "인재상이 뭐야"
  같은 질문은 Knowledge로 답한다.
- SharePoint 지원자 목록 = 평가의 "대상" 데이터.
  "김지훈 지원했어?", "승인된 지원자 목록" 같은 질문은
  실시간 데이터가 필요하다(현재는 연결된 도구가 없다).
- 문서함의 이력서 원문 = "증거". 특정 1명의 원문이 필요할 때만 링크로 연다.
```

<!-- SCREENSHOT: u3-s08 — 지침: 역할 경계 -->

{: .note }
이 블록 덕분에 3-4 시연에서 에이전트는 "기준은 알지만 데이터는 모른다"고 **스스로 경계를 드러냅니다.** 그 갭이 Unit 4에서 도구를 붙이는 동기입니다.

### 3단계. 표현 규칙

응답에 **내부 시스템 식별자**가 새어 나오지 않도록 막습니다.

```
[표현 규칙]
- 내부 도구·흐름의 식별자(예: Qv2 같은 코드명)나 시스템 내부 키를
  사용자에게 노출하지 않는다. 사람이 읽는 이름과 표현만 쓴다.
```

<!-- SCREENSHOT: u3-s09 — 지침: 표현 규칙 -->

{: .warning }
실제로 도구를 붙이면 내부 칩 이름(`Qv2` 등)이 응답·플랜에 노출된 사례가 있었습니다. 이 규칙을 미리 넣어 두면 Unit 4에서 그 누출을 예방합니다.

### 4단계. 저장 및 게시

지침을 **저장**합니다. 테스트 패널에서 바로 확인하려면 **게시(Publish)** 가 필요할 수 있습니다.

{: .note }
지침을 바꾼 뒤 동작이 안 바뀌면 **게시 여부**를 먼저 확인하세요. 저장만 하고 게시하지 않아 옛 동작이 유지되는 경우가 흔합니다.

## ✅ 체크포인트

- [ ] 역할·페르소나·말투가 작성돼 있습니다.
- [ ] 역할 경계(Knowledge=기준 / SP=대상 / 문서함=증거)가 들어 있습니다.
- [ ] 표현 규칙(내부 식별자 비노출)이 들어 있습니다.
- [ ] 지침이 저장(필요 시 게시)돼 있습니다.

## 핵심 정리

| 항목 | 내용 |
|---|---|
| 지침 = 행동 규칙 | 역할·말투·경계를 자연어로. |
| 역할 경계 | Knowledge=기준 / SP=대상 / 문서함=증거. |
| 표현 규칙 | 내부 식별자(Qv2 등) 비노출. |
| 살아있는 산출물 | 유닛마다 실패를 보고 블록을 덧붙인다. |
