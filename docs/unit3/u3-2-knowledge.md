## 개념

Knowledge는 에이전트가 답할 때 참고하는 **기준 문서**입니다. 이 채용 솔루션의 Knowledge는 4종으로 구성됩니다.

| 소스 | 역할 | 변경 빈도 |
|---|---|---|
| 채용 가이드(공통) | 공통 평가기준 **개념 정의** + 경력 레벨 정의 | 낮음 |
| 직무기술서(JD) | 직군별 필수/우대 요건 | 높음 (포지션마다) |
| 회사 지침 | 인재상 4축 | 거의 고정 |
| 평가 루브릭 | 위 셋을 점수로 환산하는 **채점 규칙**(메타 레이어) | 낮음 |

이 4종은 **소량·안정·전원 공유** 문서라 RAG(검색 증강 생성)의 정석 사용처입니다. 그래서 CS에 직접 업로드합니다.

{: .important }
**이력서 원문은 Knowledge에 넣지 않습니다.** 수십·수백 명의 이력서를 RAG에 올리면 ① 업로드 한도에 부딪히고 ② 검색이 **올려둔 이력서 전체 더미(코퍼스)** 에서 일어나 **동명이인·교차 오염**(다른 지원자 내용 섞임)이 필연입니다. 이력서는 **대상 데이터**라 SharePoint 목록에 두고, 원문이 필요한 순간엔 링크로 그 1건만 가져옵니다(Unit 4의 면접 질문).

{: .note }
대신 적재 흐름(Unit 1)이 만든 **이력서요약**이 일상 작업의 기본 재료입니다 — 빠르고, 목록에 늘 떠 있어 전수 스캔이 됩니다. **"요약 = 기본 / 원문 = 필요할 때만"** 이라는 2단 구조는 Unit 4에서 다시 만납니다.

{: .note }
**코퍼스(corpus)** = 검색 대상이 되는 문서 전체 더미. RAG는 이 더미 전체에서 관련 조각을 찾습니다. 그래서 이력서 수백 개를 한 더미에 넣으면 "이 한 명"만 깔끔히 보는 게 보장되지 않습니다.

{: .note }
**루브릭이 키스톤입니다.** 루브릭은 다른 셋의 내용을 복제하지 않고 참조만 합니다 — R1은 JD를, R2는 채용 가이드의 경력 레벨을, R5는 회사 지침의 인재상을 가리킵니다. "정의는 각 문서, 채점은 루브릭"이라는 분업입니다.

## 단계별 가이드

### 1단계. Knowledge 추가 화면 열기

면접관 에이전트의 **Knowledge** 탭에서 **`+ 추가`** 를 선택하고 **파일 업로드**를 고릅니다.

<!-- SCREENSHOT: u3-s04 — 에이전트 Knowledge 탭, 파일 업로드 -->

### 2단계. PDF 4종 업로드

아래 4개 PDF를 업로드합니다.

```
채용 가이드 문서.pdf
직무기술서(JD).pdf
회사 지침 문서.pdf
평가 루브릭.pdf
```

각 파일에 **설명(Description)** 을 달아 두면, 에이전트가 질문에 맞는 문서를 더 잘 고릅니다(예: 루브릭 → "지원자 적합도를 등급으로 매기는 채점 규칙").

<!-- SCREENSHOT: u3-s05 — 4종 PDF 업로드 + 설명 -->

{: .warning }
DOCX·HWP보다 **PDF**가 인덱싱이 안정적입니다. 원본이 Markdown이면 PDF로 변환해 올립니다(이 과정의 4종은 PDF가 준비돼 있습니다).

### 3단계. 인덱싱 완료 대기

업로드 후 각 문서가 **인덱싱(Ready)** 될 때까지 기다립니다. 소량 4종은 보통 수 분 내 완료됩니다.

<!-- SCREENSHOT: u3-s06 — Knowledge 4종 인덱싱 완료(Ready) 상태 -->

{: .note }
인덱싱이 끝나기 전에는 에이전트가 그 문서를 검색하지 못해 "모른다"고 답할 수 있습니다. 다음 서브유닛(지침)과 시연(3-4)은 **4종이 모두 Ready 된 뒤** 진행하세요. 강의에서는 이 대기 시간을 지침 작성과 겹쳐 흡수합니다.

---

## ✅ 체크포인트

- [ ] Knowledge에 PDF **4종**이 업로드돼 있습니다.
- [ ] 각 문서에 역할을 설명하는 **설명**이 달려 있습니다.
- [ ] 4종이 모두 **인덱싱 완료(Ready)** 상태입니다.
- [ ] 이력서 원문은 Knowledge에 **없습니다**.
