# Unit 3. 에이전트 + Knowledge + 지침 🤖

드디어 **면접관 에이전트**가 등장합니다. 에이전트를 생성하고, 평가의 기준이 되는 Knowledge 4종 — 채용 가이드, 직무기술서(JD), 회사 지침, 평가 루브릭 — 을 등록한 뒤, 에이전트의 행동 규칙(지침)을 작성합니다.

이 유닛의 백미는 **Knowledge-only 시연**입니다. "백엔드 채용 기준이 뭐야?"에는 술술 답하던 에이전트가 "김지훈 지원자 상태 알려줘"에는 **답하지 못하는** 순간을 직접 목격합니다. 기준(Knowledge)은 알지만 데이터(SharePoint)는 모른다 — 이 갭이 다음 유닛에서 도구를 연결하는 동기가 됩니다.

## 이 유닛을 마치면

- 면접관 에이전트가 생성되고 Knowledge 4종이 인덱싱되어 있습니다.
- 질문 5단계 시연으로 Knowledge가 답할 수 있는 범위를 확인합니다.
- "Knowledge = 기준, SP 리스트 = 대상 데이터"라는 역할 경계가 체득됩니다.

{: .important }
이력서 원문은 Knowledge에 **넣지 않습니다**. 수십·수백 명의 이력서를 RAG에 넣으면 동명이인·교차 오염이 필연입니다. 원문이 필요한 순간엔 링크로 그 1건만 가져옵니다 — 왜 그런지 본문에서 설계 논리를 다룹니다.
