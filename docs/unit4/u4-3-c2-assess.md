## 🎯 학습 목표

- 지원자의 **이력서요약**을 **평가 루브릭**에 대조해 적합도를 산출한다.
- 루브릭의 **5축 + 정성 밴드** 채점 방식을 이해한다.
- 평가 품질이 **요약 품질에 의존**한다는 한계(수치 누락)를 안다.
- 기본은 요약, 원문은 필요할 때만 부르는 **2단 구조**와 그 이유를 안다.

## 개념

적합도 평가(C2)는 **새 흐름이 아닙니다.** 조회로 가져온 지원자의 **이력서요약**(SharePoint)과 **평가 루브릭**(Knowledge)을 에이전트가 대조해 등급을 내는 것입니다. 같은 도구 + Knowledge + 지침의 조합입니다.

루브릭은 5개 축으로 보고, 산술 합산이 아니라 **정성 밴드**로 종합합니다.

| 축 | 본다 |
|---|---|
| R1 직군 요건 〔게이트〕 | 필수 요건 충족 여부 (JD) |
| R2 경력 레벨 | 연차·레벨 적합성 |
| R3 성과·임팩트 | 정량 성과 신호 |
| R4 협업·소통 | 팀워크 근거 |
| R5 인재상 부합 | 회사 인재상 4축 |

{: .note }
R1은 **게이트 축**입니다. 필수 요건을 못 맞추면 다른 축이 좋아도 등급이 올라가지 않습니다. 등급은 "대부분 충족 → 적극 추천"처럼 정성적으로 종합합니다(LLM이 산수 일관성에 약해서 의도적으로 산술을 뺐습니다).

## 단계별 가이드

### 1단계. 평가 지침 추가

지침에 평가의 입력·기준·출력 형식을 명시합니다.

```
[적합도 평가]
- 평가 대상의 근거는 지원자의 "이력서요약"(SharePoint)이다.
- 채점 기준은 "평가 루브릭"(Knowledge)을 따른다 — R1~R5 5축.
- R1(직군 요건)은 게이트다: 필수 요건 미충족이면 상위 등급을 주지 않는다.
- 결과는 등급 + 축별 근거(강/중/약)로 정성적으로 제시한다.
```

### 2단계. 테스트 — 특정 지원자 평가

"김지훈 적합도 평가해줘"처럼 물어, 에이전트가 요약↔루브릭을 대조해 **등급 + 축별 근거**를 내는지 확인합니다.

<!-- SCREENSHOT: u4-s09 — 적합도 평가 결과(등급 + 축별 근거) -->

### 3단계. 한계 관찰 — 요약에 수치가 없으면 약해진다

성과·임팩트(R3)를 평가할 때, 이력서 원문엔 "산업재해 30% 감소" 같은 수치가 있어도 **요약에서 그 수치가 빠졌으면** 그 축이 약하게 매겨집니다.

{: .important }
**등급이 지원자 실력이 아니라 "요약에 수치가 살아남았는지"를 추적하는 셈입니다.** 그래서 적재 흐름(Unit 1)의 수치 보존과 승인 흐름(Unit 2)의 요약 품질 검증이 여기서 빛을 봅니다 — C2의 평가 재료가 곧 그 요약입니다.

### 4단계. 원문 에스컬레이션 — 요약 기본, 원문은 필요할 때만 (2단 구조)

그럼 "원문이 더 깊은데 왜 요약으로 평가하나?"라는 의문이 듭니다. 답은 **둘이 대체재가 아니라 2단 구조**라는 것입니다.

| 층 | 무엇 | 언제 |
|---|---|---|
| **요약**(목록 컬럼) | 빠르고·싸고·사람이 승인한 작업층 | 조회·1차 적합도 — **기본, 전수** |
| **원문**(파일 fetch) | 느리고·비싼 정밀층 | 정밀 확인·요청 시 — **필요할 때만** |

요약을 기본으로 두는 이유는 분명합니다 — **빠르고**(원문 읽기는 로딩이 꽤 깁니다), 목록에 늘 떠 있어 **전수 스캔**이 되고, 승인 흐름에서 **사람이 품질을 검증**한 산출물이기 때문입니다. 그래서 매 평가마다 원문을 부르지 않고, 사용자가 원할 때만 부르도록 지침으로 단계를 나눕니다.

```
[적합도 평가·면접 질문 — 근거 단계]
- 기본은 이력서요약(목록)을 근거로 빠르게 평가한다.
- 사용자가 "원문 / 정밀 / 자세히 / 이력서 직접 확인"을 요청할 때만
  '경로사용 파일콘텐츠' 도구로 원문을 가져와 분석한다.
- 요약만으로 수치·세부가 부족해 단정이 어려우면, 단정하지 말고
  "원문을 확인하면 더 정밀합니다"라고 안내하며 원문 조회를 제안한다.
```

<!-- SCREENSHOT: u4-s09b — 요약 기반 vs 원문 기반 평가 결과 비교 -->

{: .important }
**실측 비교** — 같은 지원자를 요약만으로 / 원문까지 평가한 결과:

| 평가 축 | 요약 기반 | 원문 기반 |
|---|---|---|
| 경력 레벨 | 강 | 강 |
| 성과·임팩트 | **중** | **강** (수치 "30%↓" 확인) |
| 종합 등급 | 추천 | 추천 |

**종합 등급은 같습니다** — 요약만으로도 1차 판정엔 충분하다는 뜻입니다. 원문은 특정 축(성과·임팩트)을 **정밀 보정**할 때 값을 더합니다. "싸고 넓게(요약) 먼저, 비싸고 깊게(원문)는 가려서"가 되는 거죠.

{: .note }
마지막 지침 줄이 묘수입니다 — 요약의 한계(수치 누락)에 부딪히면 에이전트가 **스스로 원문 조회를 권유**합니다. 요약의 약점을 "원문 더 볼까요?"라는 기능으로 바꾸는 셈입니다.

## ✅ 체크포인트

- [ ] 특정 지원자에 대해 **등급 + 축별 근거**가 나옵니다.
- [ ] R1 미충족 지원자가 상위 등급을 받지 않습니다(게이트 동작).
- [ ] 평가 근거가 이력서요약 + 루브릭에 입각해 있습니다.

## 핵심 정리

| 항목 | 내용 |
|---|---|
| C2 = 대조 | 이력서요약(SP) ↔ 루브릭(Knowledge). 새 흐름 아님. |
| 5축 + 정성 밴드 | R1 게이트 + R2~R5, 산술 아닌 정성 종합. |
| 요약 의존 | 평가 품질 = 요약 품질. 수치 누락 시 R3 약화. |
| 2단 구조 | 기본 요약(빠름) / 원문은 요청 시(정밀). 종합 등급은 요약만으로도 대개 충분. |
| 파이프라인 | Unit 1 수치 보존 → Unit 2 품질 검증 → C2 평가로 연결. |
