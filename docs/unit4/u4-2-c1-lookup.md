## 🎯 학습 목표

- 커넥터로 지원자를 조회하고, **지침만으로** 승인된 지원자만 보이게 한다.
- 게이트의 견고함이 도구가 아니라 **지침에서 나온다**는 것을 체험한다.
- 쉬운 길(커넥터+지침)의 한계와, 더 단단한 길(흐름 강제 필터)의 차이를 안다.

## 준비물

- 4-1에서 연결한 **항목 가져오기** 도구
- 승인됨/거부됨/보류 중이 **섞여 있는** 지원자 데이터(테스트용)

---

## 개념

커넥터 **항목 가져오기**는 목록에서 항목을 가져옵니다 — 조건을 따로 걸지 않으면 **사실상 전부** 가져옵니다. 그렇다면 "승인된 사람만 보여줘"는 누가 할까요? **지침**입니다.

| 단계 | 누가 | 무엇을 |
|---|---|---|
| 가져오기 | 커넥터 | 목록 항목을 (사실상 전부) 읽어 옴 |
| 거르기·표시 | 지침 + 오케스트레이터 | 승인된 지원자만 골라 대화에 표시 |

즉 대화창에는 승인된 사람만 뜨지만, **속으로는 다 가져온 뒤 지침이 걸러 보여주는** 구조입니다. 그래서 이 유닛의 진짜 주제는 "값을 확인해 분기하는 로직"이 아니라 **"지침이 만들어내는 견고함"** 입니다. 도구는 단순한 조회기일 뿐이고, 똑똑함은 지침에 있습니다.

---

## 단계별 가이드 — 지침으로 게이트 만들기

### 1단계. 그냥 조회해 본다 (게이트 전)

지침에 아직 게이트가 없는 상태에서 "지원자 보여줘"를 물어봅니다. 거부·보류 지원자까지 섞여 나오는 것을 관찰합니다.

<!-- SCREENSHOT: u4-s04 — 게이트 전: 비승인 지원자까지 섞여 노출 -->

### 2단계. 승인 게이트 + 비노출 지침 추가

지침에 "기본 조회는 승인된 지원자만"이라는 규칙을 넣습니다.

```
[조회 기본 규칙 — 승인 게이트]
- 기본 조회는 승인상태가 '승인됨'인 지원자만 보여준다.
- 거부됨·보류 중 지원자는 목록에 포함하지도, 이름·요약을 언급하지도 않는다.
- 승인된 지원자가 0명이면 다른 상태를 나열하지 말고
  "현재 승인된 지원자가 없습니다"라고만 답한다.
- 승인상태(승인 게이트)와 검토상태(검토중/1차합격/…)는 다른 컬럼이다.
  혼동하지 않는다.
```

<!-- SCREENSHOT: u4-s05 — 지침: 승인 게이트 + 비노출 규칙 -->

{: .note }
커넥터는 데이터를 **날것**으로 돌려줍니다 — 승인 여부가 사람이 읽기 좋은 라벨이 아니라 코드값으로 올 때도 있습니다. 그래서 지침에 "승인된 상태만"이라는 기준과, 필요하면 그 코드를 어떻게 읽는지 **한 줄**을 더해 둡니다. **커넥터는 날것을 주고, 지침이 읽는 법을 알려준다** — 이게 이 모듈의 핵심 감각입니다.

### 3단계. 재테스트 — 견고함 확인

"지원자 보여줘", "지금 누구 있어?"처럼 **표현을 바꿔** 물어도 승인된 사람만 **일관되게** 나오는지 확인합니다.

<!-- SCREENSHOT: u4-s06 — 게이트 후: 승인된 지원자만 일관 노출 -->

이 변화를 표로 보면, 지침 한 블록이 행동 하나를 바꾼 것이 또렷합니다.

| 지침 블록 | 막는 실패 | 지표 (행동 변화) |
|---|---|---|
| 승인 게이트 + 비노출 | 거부·보류자가 섞여 노출 | 비승인 노출 → **0** |
| 없는 정보 방어 | 데이터에 없는 걸 지어냄 | 환각 답변 → **0** |
| 표현 일관 | 같은 질문에 답이 제각각 | 표현 바꿔도 **동일** |

{: .note }
**지침 한 블록 = 행동 한 변화.** 견고함은 도구가 아니라 이렇게 지침이 쌓이며 자랍니다. 여기서 더한 승인 게이트·비노출 규칙이 바로 4-1에서 본 **가드레일** — 에이전트가 길 밖으로 못 나가게 하는 난간입니다. 가드레일을 한 줄씩 세우고(테스트→어긋남→규칙 추가→재테스트), 그렇게 "지침에 의한 견고성"이 쌓입니다.

---

## 알아두기 — 이 방식의 한계와 더 단단한 길

{: .important }
**대화에는 승인된 사람만 보이지만, 실제로는 전부 가져온 뒤 지침이 걸러 보여주는 것**입니다. 즉 차단이 "표시 단계"에서 일어납니다. 그래서 조회 대상이 아주 많아지거나 지침이 바뀌면, 비승인 지원자가 새어 보이는 **엣지케이스에 취약**할 수 있습니다. 시티즌 개발 수준의 일상적 사용에는 충분히 잘 동작하지만, 이 성질은 알고 있어야 합니다.

{: .note }
**(참고·선택) 더 단단하게 막으려면** — 모델이 비승인 지원자를 **애초에 보지도 못하게** 하려면, 조회를 흐름으로 만들어 **가져올 때부터 승인된 항목만** 걸러 옵니다(강제 필터). 그러면 비승인 데이터가 모델에 닿지 않아 엣지케이스에 더 강합니다. 대신 흐름을 따로 만들어야 하고, 이 과정의 쉬운 길(커넥터+지침)에서는 다루지 않습니다 — "규정상 절대 노출 금지" 같은 요구가 있을 때의 선택지로 알아 두세요.

## ✅ 체크포인트

- [ ] 커넥터로 지원자를 조회할 수 있습니다.
- [ ] 기본 조회에 **승인된 지원자만** 나옵니다.
- [ ] 거부·보류 지원자가 보이지 않습니다.
- [ ] 표현을 바꿔 물어도 결과가 **일관**됩니다.
- [ ] (이해) 대화엔 승인됨만 보여도 **내부는 다 가져온 것**임을 안다.

## 핵심 정리

| 항목 | 내용 |
|---|---|
| 커넥터 = 조회기 | 조건 없으면 다 가져옴. 거르는 일은 지침이 한다. |
| 견고함의 출처 | 도구가 아니라 **지침**. 블록 하나가 행동 하나를 바꾼다. |
| 쉬운 길의 한계 | 표시 단계 차단이라 엣지케이스에 취약(일상 사용엔 충분). |
| 더 단단한 길(참고) | 흐름 강제 필터 = 모델이 비승인을 애초에 못 봄(선택). |
