## 개념

흐름이 전제조건·유일성·원자성 같은 **하드 통제**를 맡았다면, 사람의 **최종 의사결정**은 대화에서 받습니다. "1차 합격 처리해줘"에 곧장 실행하지 않고, **"정말 할까요?"** 를 한 번 묻는 것 — 이것이 **대화형 HITL**입니다.

이 확인은 흐름이 아니라 **지침(가드레일)** 이 맡습니다. 즉 역할이 나뉩니다.

| 층 | 무엇 | 담당 |
|---|---|---|
| 하드 | 전제조건·유일성·원자성·불변식 | 흐름(5-2·5-3) |
| 소프트 | 실행 전 확인·위험 요청 거절 | 지침(이 서브유닛) |

{: .note }
이번엔 가장 **단순한 텍스트 확인**으로 만듭니다. 챗봇이 평범한 문장으로 "정말 할까요?"를 묻는 baseline이죠. 다음 서브유닛에서 같은 확인을 **적응형 카드**로 끌어올려, "같은 HITL도 결정성·UX가 어떻게 달라지는지" 비교합니다.

{: .important }
**Unit 2의 비동기 HITL과 비교해 보세요.** 둘 다 "사람을 끼워 넣는" 것이지만 자리가 다릅니다.

| | 비동기 HITL (Unit 2) | 대화형 HITL (이 유닛) |
|---|---|---|
| 방식 | SharePoint 내장 승인 카드 | 챗봇 대화 중 확인 질문 |
| 트리거 | 항목 생성 (자동 이벤트) | 사용자 요청 (온디맨드) |
| 목적 | AI 요약 품질 검증 | 면접 확정 의사결정 |
| 확인 통제 | 흐름이 빈값 방어 | 지침이 확인을 강제 |

## 단계별 가이드

### 1단계. 쓰기 안전 가드레일 추가

지침에 검토상태 변경의 안전 규칙을 넣습니다.

```
[검토상태 변경 — 쓰기 안전]
- 검토상태 변경은 '검토상태 변경' 도구로만 수행한다.
- 변경 실행 전 반드시 "○○ 지원자의 검토상태를 △△(으)로 변경할까요?"라고
  확인을 받고, 사용자가 명확히 동의한 경우에만 실제 변경을 실행한다.
- 한 번에 여러 명 또는 전원 일괄 변경 요청은 실행하지 않는다.
  위험하므로 거절하고, 대상을 한 명씩 명시해 달라고 요청한다.
- 검토상태 변경 도구로 승인상태를 바꾸려 시도하지 않는다.
```

<!-- SCREENSHOT: u5-s11 — 지침: 쓰기 안전 가드레일 -->

### 2단계. 테스트 — 확인 게이트

"김지훈 1차 합격 처리해줘"라고 합니다. 에이전트가 곧장 바꾸지 않고 **"김지훈 지원자의 검토상태를 1차합격으로 변경할까요?"** 라고 되묻는지, 동의해야 실제로 흐름이 실행되는지 확인합니다.

<!-- SCREENSHOT: u5-s12 — 변경 전 확인 질문 → 동의 후 실행 -->

### 3단계. 테스트 — 위험 요청 거절(2층 방어)

"전원 불합격으로 바꿔줘" 같은 일괄 요청을 던집니다. 에이전트가 **실행하지 않고 거절**하며 한 명씩 지정해 달라고 안내하는지 확인합니다.

<!-- SCREENSHOT: u5-s13 — 일괄 변경 거절 -->

{: .note }
일괄 요청은 사실 **두 겹**으로 막힙니다. ① **흐름(하드)** — 두 이름을 한 입력에 뭉치면 조회 0건 → 행 갯수 게이트가 차단. ② **지침(소프트)** — "여러 명/전원 일괄은 거절, 한 명씩"이 먼저 가로챔. 절대 어기면 안 되는 불변식은 흐름이, 맥락에 따라 유연해야 하는 확인·거절은 지침이 — 두 층이 함께 가야 안전합니다.

---

## ✅ 체크포인트

- [ ] 검토상태 변경 전에 에이전트가 **확인 질문**을 합니다.
- [ ] 동의했을 때만 흐름이 실행됩니다.
- [ ] 일괄/전원 변경 요청을 **거절**합니다.
- [ ] 검토상태 변경 도구로 승인상태를 바꾸려 하지 않습니다.

---

## 핵심 정리

| 항목 | 내용 |
|---|---|
| 대화형 HITL | 실행 직전 "정말 할까요?"를 대화로 확인. |
| 소프트/하드 분담 | 확인·거절 = 지침 / 전제조건·유일성·원자성 = 흐름. |
| 2층 방어 | 일괄 요청 = 흐름 단건계약(하드) + 지침 거절(소프트). |
| baseline | 텍스트 확인 — 다음 유닛에서 카드로 업그레이드. |
