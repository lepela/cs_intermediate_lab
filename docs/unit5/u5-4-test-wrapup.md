## 🎯 학습 목표

- 면접 확정 트랜잭션을 **전 과정 실행**해 정상·차단·엣지를 검증한다.
- 지침으로 막는 것과 못 막는 것(서버 부작용)의 **경계를 확인**한다.
- 과정 전체의 결론 — **읽기는 커넥터, 상태 변경은 흐름** — 을 회고한다.

## 단계별 가이드

### 1단계. 정상 경로 — 끝까지 실행

승인되고 적합도 평가까지 된 지원자에게 "1차 합격 처리해줘"를 합니다.

- 확인 질문 → 동의 → 흐름 실행 → 검토상태·종속 컬럼이 함께 갱신됐는지 목록에서 확인.

<!-- SCREENSHOT: u5-s10 — 정상 경로: 확인 후 갱신 완료 -->

### 2단계. 차단 경로 — 전제조건 미충족

아직 승인 안 됐거나 적합도 평가 전인 지원자에게 같은 요청을 합니다. 흐름이 **전제조건에서 막고** 사유를 돌려주는지 확인합니다.

<!-- SCREENSHOT: u5-s11 — 차단 경로: 전제조건 미충족 거부 -->

### 3단계. 엣지 + 서버 부작용 확인

- **일괄 변경** 요청 → 거절되는가(지침).
- **승인상태 직접 변경** 시도 → 거절되는가(지침).
- **moderation 리셋** → 검토상태를 바꾼 뒤 그 지원자의 승인상태가 어떻게 됐는지 본다. 흐름이 정리하지 않으면 보류중으로 돌아가 조회에서 사라질 수 있음을 확인.

<!-- SCREENSHOT: u5-s12 — moderation 리셋 동작 확인 -->

{: .note }
이 마지막 확인이 과정의 핵심 교훈을 손으로 만지게 합니다 — **지침은 "행동"을 막지만, 시스템이 일으키는 부작용은 흐름만 정리할 수 있다.**

## ✅ 체크포인트

- [ ] 정상 경로: 확인 → 흐름 실행 → 다중 컬럼 갱신 완료.
- [ ] 차단 경로: 전제조건 미충족 시 갱신되지 않음.
- [ ] 일괄·승인상태 변경이 거절됨.
- [ ] moderation 리셋 동작을 직접 확인함.

## 🎓 마무리 회고 — 이 과정이 남긴 한 문장

여기까지 두 개의 풀 빌드(적재 흐름·면접 확정 흐름)와 한 개의 커넥터 기반 모듈(조회·평가·질문)을 만들었습니다. 그 전부를 관통하는 결론은 하나입니다.

> **읽기는 커넥터 + 지침으로, 상태를 바꾸는 일은 흐름으로.**

| 경계 | 쉬운 길 | 단단한 길 |
|---|---|---|
| 읽기·조회·평가 | 커넥터 + 지침(가드레일) | — |
| 상태 변경·트랜잭션 | — | 흐름(전제조건·원자성·불변식) |
| 사람 끼워 넣기 | 비동기 HITL(이벤트) | 대화형 HITL(온디맨드) |
| 통제 | 소프트(지침) — 행동을 좁힘 | 하드(흐름·시스템) — 부작용·불변식 |

이 과정의 진짜 기술은 도구 사용법이 아니라 **"언제 커넥터로 충분하고, 언제 흐름이 필요한가"를 판단하는 것**입니다. 모델이 좋아질수록 조회·추론은 생성형 오케스트레이션이 흡수하고, 흐름은 결정성·게이트·이벤트가 필요한 핵심으로 남습니다. 그 경계를 손으로 그어 본 것 — 그게 이 강의가 남기는 것입니다.
