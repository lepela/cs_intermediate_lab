## 개념

실제 구조는 **3레이어**입니다. 오케스트레이터(지침) → 토픽(게이트·카드) → 흐름(갱신)이 역할을 나눠 맡습니다.

```
"1차 합격 처리해줘"
  → [오케스트레이터 지침] 지원자 조회 + AI적합도 확인
       → 미적용이면 루브릭 교차 평가 → 등급 결정(자동)
  → [토픽: 검토상태 변경 확인] 호출
       → 게이트 ① count=1 (0건→없음·2건↑→동명이인 차단)
       → 게이트 ② 승인됨 확인 (미승인 차단)
       → 게이트 ③ AI적합도 ≠ 미적용 확인 (평가 전 차단)
       → 적응형 카드로 확인 → 사람이 확정
  → [흐름: 상태변경 흐름] 다중 컬럼 원자 갱신 → 결과 반환
```

핵심은 **불변식을 어디에 가두느냐**입니다. "승인도 안 된 사람을 1차합격시키지 않는다", "검토상태와 AI적합도를 함께 맞춘다" 같은 규칙을 확률적 LLM이 아니라 **토픽 게이트와 흐름이 결정적으로** 지킵니다.

---

## 단계별 가이드

### 1단계. 흐름 만들기 + 입력 정의

CS에서 새 흐름 **`검토상태 변경`** 을 만들고, 트리거를 **"에이전트가 흐름을 호출할 때"** 로 둡니다. 입력은 세 가지입니다.

| 입력 | 용도 |
|---|---|
| 지원자이름 | 대상 지원자(예: 김지훈) |
| 검토상태 | 목표 상태 — 1차합격 / 보류 / 불합격 / 검토중 |
| AI적합도 | 보조 입력(전제조건·로깅용) |

<!-- SCREENSHOT: u5-s03 — 흐름 트리거 + 입력 3개 정의 -->

### 2단계. 입력에 설명 달기 — 슬롯 필링 교정

입력만 만들어 두고 테스트하면, 에이전트가 값을 **엉뚱한 칸에** 채우는 일이 생깁니다. "김지훈 1차합격으로 변경"이라고 했는데 검토상태 칸에 "김지훈"이, 적합도 칸에 "1차합격"이 들어가는 식입니다. 입력의 **설명이 비어 있어** 오케스트레이터가 어느 값이 어디로 가는지 헷갈리기 때문입니다.

각 입력의 **추가 세부사항 → 설명**에 다음을 적어 줍니다.

| 입력 | 설명(슬롯 필링 힌트) |
|---|---|
| 지원자이름 | "대상 지원자 이름만 그대로(예: 김지훈). 상태·적합도 값을 넣지 말 것." |
| 검토상태 | "목표 상태. 1차합격 / 보류 / 불합격 / 검토중 중 하나." |
| AI적합도 | "보조 입력. 이름·상태를 넣지 말 것. 불분명하면 빈 값." |

<!-- SCREENSHOT: u5-s04 — 입력별 설명(슬롯 필링) 작성 -->

{: .note }
흐름 자체는 한 줄도 고치지 않고 **도구 레벨의 설명만으로** 값 매핑을 바로잡았습니다. 입력 설명은 오케스트레이터에게 주는 "이 칸엔 이런 값"이라는 힌트입니다 — Unit 4의 도구 설명이 라우팅을 좌우했던 것과 같은 원리입니다.

### 3단계. 토픽 만들기 — 게이트 + 카드를 담는 그릇

전제조건 검사와 사람 확인은 **토픽**이 담당합니다. `검토상태 변경 확인`이라는 이름으로 새 토픽을 만들고, 오케스트레이터가 이 토픽을 호출하면 다음 흐름으로 진행합니다.

```
토픽 진입
  → 게이트 ① : 조회 count = 1? (동명이인·없음 차단)
  → 게이트 ② : 승인상태 = 승인됨? (미승인 차단)
  → 게이트 ③ : AI적합도 ≠ 미적용? (평가 전 차단)
  → 카드 표시 → 사람 확인
  → 확정이면 갱신 흐름 호출 / 취소면 중단
```

<!-- SCREENSHOT: u5-s05 — 토픽 게이트 3층 구조 -->

{: .important }
게이트가 **토픽 안에** 있다는 게 핵심입니다. 커넥터+지침이라면 "확률적으로 대개 지킴"이지만, 토픽은 **조건을 못 넘으면 절대 다음 노드로 가지 않습니다**(결정적). "1차합격은 평가가 끝난 승인된 사람만"이라는 불변식을 시스템이 강제합니다.

### 4단계. 흐름 만들기 — 원자 갱신만 담당

게이트를 통과하고 사람이 확인하면, 토픽이 **상태변경 흐름**을 호출합니다. 이 흐름의 역할은 단순합니다 — 종속 컬럼들을 한 번에 묶어 갱신합니다.

| 컬럼 | 갱신 |
|---|---|
| 검토상태 | 목표 값(예: 1차합격) |
| AI적합도 | 오케스트레이터가 평가한 등급(미적용이었으면 채워서 전달) |

<!-- SCREENSHOT: u5-s06 — 상태변경 흐름 항목 업데이트 -->

{: .note }
컬럼들이 **서로 종속**돼 있어 따로따로 바꾸면 데이터가 어긋납니다. 흐름은 이들을 **한 묶음**으로 갱신해 "전부 바뀌거나 전부 안 바뀌거나"를 보장합니다 — 이게 트랜잭션의 원자성입니다.

### 5단계. 흐름을 에이전트 도구로 붙이기

상태변경 흐름을 면접관 에이전트의 **도구**로 추가합니다. 도구 설명은 라우팅을 좌우하므로 "지원자의 검토상태를 1차합격/보류/불합격/검토중으로 변경할 때 호출"처럼 명확히 적습니다.

<!-- SCREENSHOT: u5-s07 — 흐름을 에이전트 도구로 추가 -->

{: .warning }
지금 흐름은 **이름으로 조회**합니다. 하지만 동명이인이 있으면? 이름이 여러 건이거나 0건이면? 아직 그 방어가 없습니다 — 다음 서브유닛에서 **대상을 유일하게 특정**하도록 하드닝합니다.

---

## ✅ 체크포인트

- [ ] **검토상태 변경 확인** 토픽이 게이트 3층(count=1·승인됨·적합도≠미적용)을 갖춥니다.
- [ ] **상태변경 흐름**이 만들어져 입력에 **설명**을 달아 값 매핑이 정확합니다.
- [ ] 토픽이 확인 후 흐름을 호출하고, 흐름이 **여러 컬럼을 한 번에** 갱신합니다.
- [ ] 흐름이 에이전트의 **도구**로 붙어 있습니다.

## 핵심 정리

| 항목 | 내용 |
|---|---|
| 3레이어 분담 | 오케스트레이터(조회·평가) → 토픽(게이트·카드) → 흐름(갱신). |
| 토픽 게이트 | count=1·승인됨·적합도≠미적용 3층 — 못 넘으면 절대 갱신 없음. |
| 입력 설명 | 슬롯 필링 힌트로 값이 엉뚱한 칸에 들어가는 것 방지. |
| 원자 갱신 | 종속 컬럼을 한 묶음으로 — 전부 아니면 전무. |
