## 단계별 가이드 — 4가지 시나리오

흐름이 단단한지 확인하는 가장 좋은 방법은 **일부러 엣지를 던지는 것**입니다. 네 가지를 차례로 봅니다.

### 1단계. 정상 경로 — 미적용 포함 자동 평가 후 1차합격

승인됐지만 **AI적합도가 미적용**인 지원자에게 "1차 합격 처리해줘"를 합니다.

```
에이전트 동작 순서:
  1. 지원자 조회 → 현재 상태 서술(검토중, 미적용)
  2. AI적합도 = 미적용 감지
  3. 루브릭 교차 평가 → 적극추천/추천/면접확인필요/부적합 판정
  4. 토픽 호출 (AI적합도 함께 전달)
  5. 카드 표시: 이름 / 현재검토상태 / AI적합도(평가 결과) / 변경상태 선택지
  6. 사람이 확정 → 상태변경 흐름 → AI적합도·검토상태 동시 갱신
```

- 갱신 후 조회로 **검토상태·AI적합도** 두 컬럼이 함께 바뀌었는지 확인.

<!-- SCREENSHOT: u5-s18 — 정상 경로: 미적용→AI평가→카드→확정→갱신 완료 -->

### 2단계. 없는 이름 — 0건

**존재하지 않는 이름**으로 요청합니다. 토픽 게이트 ①에서 **count=0**으로 차단하고 "그 이름의 지원자가 없습니다"를 돌려주는지 확인합니다(갱신 X).

<!-- SCREENSHOT: u5-s19 — 0건: 없는 이름 차단 -->

### 3단계. 전제조건 미충족 — 승인 안 된 지원자

**아직 승인되지 않은**(승인상태 ≠ 승인됨) 지원자에게 변경 요청을 합니다. 토픽 게이트 ②에서 차단하고 "AI 요약 승인이 완료되지 않은 지원자"를 안내하는지 확인합니다.

<!-- SCREENSHOT: u5-s20 — 게이트 ②: 미승인 차단 -->

### 4단계. 배치 + 서버 부작용

- **일괄 변경** "전원 불합격으로 바꿔줘" → 오케스트레이터 지침이 거절하는가.
- **승인상태 직접 변경** 시도 → 거절되는가(지침).
- **moderation 리셋** — 검토상태를 바꾼 뒤 그 지원자의 승인상태를 확인한다. 보류중으로 돌아가 기본 조회(승인됨만)에서 사라지는 SP 서버 부작용을 직접 목격한다.

<!-- SCREENSHOT: u5-s21 — 배치 거절 / moderation 리셋 확인 -->

{: .note }
이 마지막 확인이 과정의 핵심 교훈을 손으로 만지게 합니다 — **지침은 "행동"을 막지만, 시스템이 일으키는 부작용은 흐름만 정리할 수 있다.**

## 🎓 마무리 회고 — 이 과정이 남긴 한 문장

여기까지 두 개의 풀 빌드(적재 흐름·면접 확정 흐름)와 한 개의 커넥터 기반 모듈(조회·평가·질문)을 만들었습니다. 그 전부를 관통하는 결론은 하나입니다.

> **읽기는 커넥터 + 지침으로, 상태를 바꾸는 일은 흐름으로.**

| 경계 | 쉬운 길 | 단단한 길 |
|---|---|---|
| 읽기·조회·평가 | 커넥터 + 지침(가드레일) | — |
| 상태 변경·트랜잭션 | — | 흐름(전제조건·유일성·원자성·불변식) |
| 사람 끼워 넣기 | 비동기 HITL(이벤트) | 대화형 HITL(온디맨드) |
| 대화형 확인 | 텍스트(유연·모호) | 적응형 카드(결정적·선택) |
| 통제 | 소프트(지침) — 행동을 좁힘 | 하드(흐름·시스템) — 부작용·불변식 |

같은 HITL도 **결정성과 UX의 트레이드오프**가 다릅니다. 텍스트 확인은 빠르고 유연하지만 응답이 모호하고, 카드 확인은 한 겹 더 만들어야 하지만 버튼 값으로 또렷하게 분기합니다. "언제 어느 쪽이 맞나"를 고르는 것도 이 과정이 가르친 판단입니다.

이 과정의 진짜 기술은 도구 사용법이 아니라 **"언제 커넥터로 충분하고, 언제 흐름이 필요한가"를 판단하는 것**입니다. 모델이 좋아질수록 조회·추론은 생성형 오케스트레이션이 흡수하고, 흐름은 결정성·게이트·이벤트가 필요한 핵심으로 남습니다. 그 경계를 손으로 그어 본 것 — 그게 이 강의가 남기는 것입니다.
