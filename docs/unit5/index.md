# Unit 5. 모듈 2 — 면접 확정 트랜잭션 ⚖️

마지막 유닛이자 두 번째 *풀 빌드*{: .label .label-purple } 산출물. "이 지원자 1차 합격 처리해줘"라는 한 마디 뒤에는 전제조건 검사(승인이 끝난 지원자인가?), 서로 종속된 3개 컬럼의 원자적 갱신, 그리고 실행 직전 사람의 최종 확인(**대화형 HITL**, Human-in-the-Loop)이 숨어 있습니다. 이런 **상태 변경 트랜잭션**은 확률적으로 움직이는 LLM에게 맡길 수 없습니다 — 결정적 흐름으로 만듭니다. (설계 문서에서는 이 흐름을 **F-C**라는 코드명으로 부릅니다.)

Unit 4에서 "커넥터로 이렇게까지 되네?"를 봤다면, 여기서는 **커넥터가 실패하는 엣지케이스**(전원 일괄 변경, 승인상태 직접 변경, 서버 부작용)를 먼저 목격한 뒤 흐름으로 전환합니다. 두 유닛을 합치면 이 과정의 최종 결론이 완성됩니다: **읽기는 커넥터+지침, 상태를 바꾸는 일은 흐름.**

## 이 유닛을 마치면

- 전제조건 게이트 + 대화형 확인 + 다중 컬럼 갱신이 담긴 면접 확정 흐름이 완성됩니다.
- 지침으로 막을 수 있는 것과 막을 수 없는 것(서버 부작용)의 경계를 압니다.
- 비동기 HITL(Unit 2)과 대화형 HITL(이 유닛)을 모두 만들어 본 사람이 됩니다.

{: .warning }
콘텐츠 승인이 켜진 리스트는 **항목을 수정하면 승인상태가 보류중으로 리셋**됩니다. 지침으로는 막을 수 없는 SharePoint 서버 동작 — 쓰기를 흐름에 맡겨야 하는 결정적 이유 중 하나입니다.
