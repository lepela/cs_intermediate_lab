## 🎯 학습 목표

- 텍스트 확인을 **적응형 카드(Adaptive Card)** 확인으로 끌어올린다.
- 지원자이름·현재 상태·AI적합도·**변경상태 선택지**를 **변수로 바인딩**해 카드를 런타임에 동적으로 그린다.
- 버튼이 돌려주는 **응답 값을 캡처해 분기**하고(확정/취소), 오케스트레이터가 미적용 지원자를 **자동 평가**하는 흐름을 이해한다.

## 개념

텍스트 확인("정말 할까요?")은 빠르지만 **응답이 자유롭습니다.** "응 해줘", "어 그래", "ㅇㅋ"… 모델이 대부분 알아듣지만 "반드시 예/아니오"를 보장하진 못하죠. **적응형 카드**는 이 확인을 **버튼 선택**으로 바꿔, 사람이 누른 값을 결정적으로 받아옵니다.

같은 HITL이지만 성격이 다릅니다.

| | 텍스트 확인(5-4) | 적응형 카드 확인(이 유닛) |
|---|---|---|
| 응답 | 자유 문장(유연하지만 모호) | 버튼 값(결정적) |
| 정보 표시 | 한 줄 질문 | 이름·현재상태·AI적합도·변경상태 선택지를 한눈에 |
| 검토상태 입력 | 오케스트레이터가 묻거나 발화에서 추출 | **카드 드롭다운·버튼**이 담당 |
| 분기 | 지침이 "동의" 해석 | 버튼 값으로 명확히 분기 |
| 자리 | 생성형 지침 | **토픽**(결정적 대화 레이어) |

### 카드를 만드는 세 가지 방법

| 방법 | 언제 | 비고 |
|---|---|---|
| ① **Ask with Adaptive Card** 노드 | 예/아니오·선택지 확인 | **이 유닛의 정석.** 버튼 값 캡처가 내장 |
| ② 적응형 카드 노드 + Power Fx | 변수로 동적 렌더링·조건부 표시 | 카드 모양을 더 세밀하게 통제할 때 |
| ③ AI 프롬프트로 카드 JSON 생성 | 필드가 런타임에야 정해질 때 | 고정 확인엔 과함 — 생성 결과를 Power Fx로 검증 후 렌더 권장 |

우리는 **①을 기본**으로, 변수 바인딩(②의 아이디어)을 얹어 "동적인 확인 카드"를 만듭니다.

{: .note }
적응형 카드 확인은 **토픽** 안에 둡니다 — 토픽은 결정적인 대화 레이어라, "확인 → 분기 → 흐름 호출"을 또렷하게 그릴 수 있기 때문입니다. 5-4의 텍스트 확인은 생성형 지침 레이어였지만, 카드 확인은 **기존 확인 토픽에 카드 노드를 통합**하는 방향입니다.

---

## 단계별 가이드

### 0단계. 오케스트레이터 역할 — 자동 AI 평가

카드를 그리기 전에, 오케스트레이터(지침 [5])가 먼저 일을 합니다.

```
오케스트레이터 지침 [5] 동작:
  step 1. 지원자 정보 조회 + 간략 서술
  step 2. AI적합도 확인
  step 3. 미적용이면 → ResumeSummaryAI + Knowledge 루브릭 교차 평가
              → 적극추천/추천/면접확인필요/부적합 중 하나로 AI가 직접 판단
  step 4. 평가된 값을 AI적합도로 확정 후 토픽 호출
  step 5. 토픽 호출 시 AI적합도를 함께 전달
  step 6. 검토상태는 카드 UI가 담당 — 오케스트레이터가 직접 묻지 않음
```

<!-- SCREENSHOT: u5-s14 — 오케스트레이터 지침 [5] 블록 -->

{: .important }
"미적용 지원자에게 변경 요청이 왔을 때" 에이전트가 루브릭을 꺼내 **스스로 평가하고** 등급을 확정한 뒤 토픽을 호출합니다. 사람이 적합도를 직접 지정하거나 '미적용으로 바꿔달라'는 요청은 거절합니다 — 평가는 AI의 판단 영역이기 때문입니다.

### 1단계. 토픽에 카드 노드 통합

5-2에서 만든 확인 토픽(`검토상태 변경 확인`)에 **Ask with Adaptive Card** 노드를 추가합니다. 흐름은 이렇게 이어집니다.

```
토픽 진입 (게이트 ①②③ 통과 후)
  → 적응형 카드 표시
  → [확정] 버튼 → 상태변경 흐름 호출 → 모든 토픽 종료
  → [취소] 버튼 → "변경을 취소했습니다" → 모든 토픽 종료
```

<!-- SCREENSHOT: u5-s15 — 카드 노드가 게이트 통과 후 이어지는 토픽 구조 -->

### 2단계. 카드 구성 — 변수 바인딩

카드에 변수를 꽂아 **런타임에 동적으로** 그립니다. 실제 카드는 4가지 정보를 한눈에 보여 줍니다.

| 카드 요소 | 바인딩 | 비고 |
|---|---|---|
| 지원자 이름 | `{지원자이름}` 변수 | 오케스트레이터가 조회 후 전달 |
| 현재 검토상태 | `{현재검토상태}` 변수 | SP에서 조회한 값 |
| AI적합도 | `{AI적합도}` 변수 | 오케스트레이터 평가 결과 또는 기존 값 |
| 변경상태 선택지 | 검토중 / 1차합격 / 보류 / 불합격 | **카드 안에서 사람이 직접 선택** |
| 버튼 | **[확정]** · **[취소]** | 누른 값이 변수로 반환 |

<!-- SCREENSHOT: u5-s16 — 적응형 카드 변수 바인딩 + 변경상태 선택지 -->

{: .important }
같은 카드 한 장이 지원자마다 **다른 내용**으로 그려집니다. 그리고 "어떤 상태로 바꿀지"도 카드 안에서 선택합니다 — 오케스트레이터가 검토상태를 미리 묻지 않는 이유가 바로 이것입니다. **카드가 검토상태 입력 UI를 담당**합니다.

### 3단계. 응답 값 캡처 → 분기

확정 버튼을 누르면 **선택한 변경상태 값**이 변수로 돌아옵니다. 그 값으로 분기합니다.

| 누른 버튼 | 처리 |
|---|---|
| 확정 | **상태변경 흐름 호출** (지원자ID + 변경상태 + AI적합도 전달) → 모든 토픽 종료 |
| 취소 | "변경을 취소했습니다" → 모든 토픽 종료 |

<!-- SCREENSHOT: u5-s17 — 확정/취소 분기 후 흐름 호출 노드 -->

{: .note }
텍스트 확인(5-4)에서는 모델이 동의를 **해석**했습니다. 카드는 버튼 값이 **그대로** 분기 조건이 됩니다. 사람이 무엇을 골랐는지에 모호함이 없습니다. 그리고 `모든 토픽 종료`를 달아 재귀 호출을 차단하는 것도 잊지 마세요.

## ✅ 체크포인트

- [ ] 오케스트레이터 지침 [5]에 미적용 자동 평가 step이 있고, 검토상태는 카드가 담당합니다.
- [ ] Ask with Adaptive Card에 **이름·현재상태·AI적합도·변경상태 선택지**가 변수로 바인딩됩니다.
- [ ] 버튼 **응답 값으로 확정/취소**가 분기되고, 확정 시 **상태변경 흐름**이 호출됩니다.
- [ ] 확정·취소 분기 끝에 **모든 토픽 종료** 노드가 있어 재귀 호출이 차단됩니다.

## 핵심 정리

| 항목 | 내용 |
|---|---|
| 오케스트레이터 자동 평가 | 미적용 → 루브릭 교차 평가 → 등급 확정 후 토픽 호출. |
| 카드 = 결정적 확인 + 입력 | 변경상태 선택지가 카드 안에 — 오케스트레이터가 직접 묻지 않음. |
| 동적 렌더링 | 한 카드 + 변수 바인딩으로 지원자마다 다른 내용. |
| 응답 처리 | 확정(흐름 호출·모든 토픽 종료)/취소 분기. 재귀 차단 필수. |
