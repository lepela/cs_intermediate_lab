## 개념

Unit 4에서 **읽기**는 커넥터 하나 + 지침으로 깔끔히 됐습니다. 그럼 **쓰기**(검토상태 변경 같은 상태 변경)도 커넥터로 될까요? 한번 시도해 보면 — **안전하게는 안 됩니다.**

상태 변경은 단순한 한 칸 수정이 아니라 **트랜잭션**입니다. "이 지원자 1차 합격 처리해줘" 한 마디 뒤에 숨은 것들:

- **대상 특정** — 그 "이 지원자"가 정확히 누구인가? 동명이인은 없나?
- **전제조건** — 승인이 끝난 지원자인가? 적합도 평가는 됐나?
- **여러 컬럼의 종속** — 검토상태를 바꾸면 면접확인필요 등 다른 컬럼도 함께 맞아야 한다.
- **실행 직전 확인** — 사람이 "정말 할까요?"에 동의했나.

이걸 확률적으로 움직이는 LLM + 단순 커넥터에 맡기면 위험합니다. 이번 서브유닛에서 그 위험을 **직접 목격**하고, 다음부터 흐름으로 옮깁니다.

{: .note }
이 유닛의 흐름은 빈 실습용 리스트가 아니라 **에이전트가 실제로 쓰는 운영 리스트**(저장소 사이트의 `지원자 리스트`)를 대상으로 합니다. 화면의 컬럼 내부 이름이 교안 앞부분과 조금 다르게 보일 수 있는데(예: `AI적합도`의 내부명은 `AI_Adaptation`), **표시 이름 기준으로 따라오면** 됩니다.

## 단계별 가이드 — 커넥터로 써 보기(대조군)

### 1단계. 단순 변경은 된다

커넥터의 **항목 업데이트**로 한 지원자의 검토상태를 바꿔 봅니다. 단순한 경우는 동작합니다 — 그래서 "되네?" 싶습니다.

### 2단계. 엣지를 던져 본다 — 지침으로 막히는 것(소프트)

까다로운 요청을 던집니다.

| 요청 | 커넥터 단독 | 지침 가드레일 추가 시 |
|---|---|---|
| 없는 지원자 변경 | 거부(환각 쓰기 없음) | 거부 |
| 없는 값(예: "최종합격") | 거부 + 유효 선택지 제시 | 거부 |
| **전원 일괄 "불합격"** | **확인 없이 즉시 여러 명 변경** ❌ | 거절(확인 요구) ✅ |
| **승인상태를 직접 변경** | SharePoint가 거부(읽기 전용) | 거절 + 안내 ✅ |

<!-- SCREENSHOT: u5-s02 — 일괄 변경이 확인 없이 실행되는 위험 -->

{: .note }
일괄 변경·승인상태 변경 같은 위험은 **지침(가드레일)으로 어느 정도 막힙니다** — Unit 4에서 본 그 소프트 통제죠. 여기까지는 지침의 힘입니다.

### 3단계. 지침으로 **못 막는** 것 — 여기서 흐름이 필요해진다

문제는 지침으로도 안 되는 것들입니다.

- **대상의 유일성** — "김지훈"이 두 명이면? 커넥터+지침은 "대개" 잘 고르지만, "반드시 한 명만"을 보장하지 못합니다. 이건 다음 서브유닛(하드닝)에서 흐름이 풀 문제입니다.
- **moderation 리셋(서버 부작용)** — 콘텐츠 승인이 켜진 리스트는 **항목을 수정하는 순간 승인상태가 보류중으로 리셋**됩니다. 검토상태 한 번 바꿨을 뿐인데 그 지원자가 기본 조회(승인됨만)에서 **사라집니다.** 지침에 "리셋하지 마"라고 적을 수가 없어요 — SharePoint 서버가 하는 일이니까.
- **원자성** — 여러 컬럼을 "전부 바뀌거나, 전부 안 바뀌거나"로 묶어야 하는데, 커넥터 호출 몇 번으로는 중간에 끊기면 데이터가 어긋납니다.
- **불변식** — "1차합격은 적합도 평가가 끝난 사람만" 같은 규칙을 확률적 LLM이 매번 지킨다는 보장이 없습니다.

{: .important }
**소프트(지침)로 막는 건 '행동', 하드(흐름)로만 막는 건 '대상 유일성·서버 부작용·원자성·불변식'입니다.** 지침은 "위험한 요청을 거절"까지는 하지만, moderation 리셋처럼 시스템이 일으키는 부작용이나 "전부 아니면 전무"의 원자성은 **흐름 안에서만** 보장됩니다. 바로 이 지점이 "상태 변경 = 흐름"의 결정적 이유입니다.

## ✅ 체크포인트

- [ ] 커넥터로 단순 검토상태 변경이 되는 것을 확인했습니다.
- [ ] 일괄 변경·승인상태 변경이 **지침으로** 거절되는 것을 봤습니다.
- [ ] **대상 유일성·moderation 리셋·원자성·불변식**은 지침으로 못 막는다는 것을 이해했습니다.

## 핵심 정리

| 항목 | 내용 |
|---|---|
| 읽기 vs 쓰기 | 읽기는 커넥터로 OK, 쓰기(상태 변경)는 안전하지 않다. |
| 지침이 막는 것 | 위험한 요청 거절(일괄·없는값·승인상태) = 소프트. |
| 지침이 못 막는 것 | 대상 유일성·moderation 리셋·원자성·불변식 = 흐름(하드)만 가능. |
| 결론 | 상태 변경 트랜잭션 = 결정적 흐름. |
