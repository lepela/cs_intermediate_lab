- 거부 시 지원자에게 **사유를 담아 회신**하는 분기를 안전하게 구현한다.

## 개념

승인 액션이 끝나면 **결과(Outcome)** 값이 `Approve` 또는 `Reject`로 돌아옵니다. 이 값으로 흐름을 두 갈래로 나눕니다.

```
승인 결과?
├─ Approve → 콘텐츠 승인 상태 = 승인됨
└─ Reject  → 콘텐츠 승인 상태 = 거부됨 + 지원자에게 사유 회신 메일
```

핵심은 **승인 결과를 목록의 콘텐츠 승인 상태에 반영**하는 것입니다. 이래야 면접관 에이전트의 기본 조회(승인됨만)가 올바르게 동작합니다.

{: .important }
**승인상태는 직접 쓰는 컬럼이 아닙니다.** SharePoint 내장 콘텐츠 승인이라, 일반 "항목 업데이트"가 아니라 전용 액션(**콘텐츠 승인 상태 설정**)으로 바꿉니다. 챗봇·커넥터의 일반 업데이트로는 이 값을 못 바꿉니다(읽기 전용으로 거부됨) — 그래서 상태 변경은 흐름의 몫입니다.

## 단계별 가이드

### 1단계. 조건 분기 추가

승인 액션 다음에 **조건(Condition)** 을 추가하고, 승인 결과를 비교합니다.

| 왼쪽 | 연산 | 오른쪽 |
|---|---|---|
| 승인 결과(Outcome) | 다음과 같음 | `Approve` |

### 2단계. 예(True) — 승인됨으로 설정

**'예'** 분기에 SharePoint **`콘텐츠 승인 상태 설정`** 액션을 넣습니다.

| 항목 | 값 |
|---|---|
| 사이트 / 목록 | HR Lab / 지원자 마스터 |
| ID | 트리거 항목의 ID |
| 승인 상태 | `승인됨` (0) |

<!-- SCREENSHOT: u2-s07 — 콘텐츠 승인 상태 설정 = 승인됨 -->

### 3단계. 아니요(False) — 거부됨 + 사유 회신

**'아니요'** 분기에 두 액션을 넣습니다.

먼저 **콘텐츠 승인 상태 설정** = `거부됨`(1).

이어서 **메일 보내기(V2)** 로 지원자에게 회신합니다. 본문의 사유 자리에는 아래 식을 넣습니다.

```
@{if(empty(trim(coalesce(items('For_each')?['comments'],''))),
   '담당자가 별도 의견을 남기지 않았습니다. 자세한 사항은 본 메일 회신으로 문의해 주시기 바랍니다.',
   items('For_each')?['comments'])}
```

<!-- SCREENSHOT: u2-s08 — 거부 분기: 거부됨 설정 + 사유 폴백 메일 -->

{: .warning }
거부 사유는 **승인 결과값(`approverResponse`)이 아니라 주석(`comments`)** 을 참조해야 합니다. 결과값을 꽂으면 사유를 입력해도 메일에 "Reject"가 찍힙니다(실제로 있었던 버그). 위 식은 `comments`를 참조하면서, 비어 있으면 안내 문구로 **폴백**합니다.

{: .note }
`coalesce`는 null 방어, `trim`은 공백만 입력한 경우 방어입니다. 표준 승인은 "거부 시 사유 필수"를 강제할 수 없어서, **흐름이 빈 사유를 방어**하는 설계입니다. (대화형 HITL은 반대로 지침으로 사유를 대화에서 강제 — 두 패턴의 대비 포인트.)

{: .note }
HTML 메일에서는 Markdown 볼드(`**이름**`)가 먹지 않아 별표가 그대로 노출됩니다. 메일 본문에서는 별표를 빼고 작성하세요(승인 카드와 규칙이 다름).

## ✅ 체크포인트

- [ ] 승인 결과로 **예/아니요 분기**가 갈립니다.
- [ ] 예 분기: 콘텐츠 승인 상태가 **승인됨**으로 설정됩니다.
- [ ] 아니요 분기: **거부됨** 설정 + 사유 회신 메일이 발송됩니다.
- [ ] 거부 메일 사유가 `comments` 참조 + 빈값 폴백으로 작성돼 있습니다.

---

## 핵심 정리

| 항목 | 내용 |
|---|---|
| 결과 분기 | Outcome `Approve`/`Reject`로 두 갈래. |
| 콘텐츠 승인 상태 설정 | 승인상태는 전용 액션으로만 변경(일반 업데이트 불가). |
| 사유 폴백 | `comments` 참조 + 빈값 시 안내 문구. `approverResponse` 아님. |
| 방어적 설계 | 비동기 HITL은 UI를 못 막으니 흐름이 빈 사유를 방어. |
